## 2. 설계 결정

### 2-1. 앱 구조
별도 `support` 앱 생성. 기존 앱(accounts·learning·rewards)은 각각 분명한 도메인이 있어서 support를 끼워 넣으면 책임이 섞임.

### 2-2. 데이터 모델
```
Inquiry
├── user (FK → User, CASCADE)
├── content (TextField)           ← 문의 내용
├── created_at (auto_now_add)
├── answer (TextField, null)      ← 운영자가 DB에 직접 채움
├── answered_at (DateTimeField, null)
└── is_answer_read (bool, default=False)   ← 사용자가 앱에서 확인했는지
```
- 테이블명: `tbl_support_inquiry`
- 1문의 = 1행. answer가 null이면 대기 중, 값 있으면 답변 완료.
- 스레드·추가 메시지 없음.

### 2-3. 엔드포인트

| 메서드 | URL | 설명 |
|---|---|---|
| `POST` | `/api/support/inquiries/` | 문의 등록 |
| `GET` | `/api/support/inquiries/` | 내 문의 목록 (최신순) |
| `GET` | `/api/support/inquiries/unread-count/` | 미확인 답변 수 (뱃지용) |
| `PATCH` | `/api/support/inquiries/<id>/read/` | 답변 확인 처리 |

- 전부 인증 필요. 본인 데이터만 접근.

### 2-4. 프론트 화면 구조

**진입:** Settings → "문의하기" (ListRow) → `InquiryScreen`

**InquiryScreen** (단일 화면, 스크롤):
```
┌─────────────────────────────────┐
│  ← 문의하기                      │
├─────────────────────────────────┤
```

### 2-5. 뱃지/알림 전략

- `useUnreadInquiryCount()` 훅 → `GET /api/support/inquiries/unread-count/` 폴링.
- Settings의 "문의하기" ListRow 오른쪽에 빨간 점(·) 표시 (unread > 0일 때).
- InquiryScreen 진입 시 → 답변 있는 항목 순차 `PATCH read` → 뱃지 사라짐.
- 푸시 알림은 2차 (현재 미구현 영역).

### 2-6. 디자인 방향

실제 앱 사례(토스·Linear·당근마켓)에서 공통으로 보이는 패턴 채택:

- **카드 단위**: 문의 1건 = 카드 1장. 날짜(소형) + 문의 내용(본문) + 상태 or 답변.
- **답변 표시**: 문의 내용 아래에 구분선(`─`) + 답변 텍스트. 별도 답변 칸을 미리 만들지 않음.
- **상태 표현**: `answer == null` → "답변 준비 중" 우측 정렬 소형 텍스트(tertiary). 거창한 배지보단 조용한 힌트.
- **새 답변**: `is_answer_read == false` → 카드 좌측에 세로 강조 바(brand 색) 표시. 확인 후 사라짐.
- **작성 박스**: 하단 고정 InputBox + "보내기" 버튼. 전송 중 로딩(~1초), 성공 시 인라인 토스트. 전송 후 텍스트 초기화 + 목록 상단에 새 카드 추가.
- **빈 상태**: 문의가 없으면 일러스트 없이 "아직 문의가 없어요. 궁금한 점이 있으면 보내주세요." 한 줄 텍스트.
- **디자인 토큰**: `tokens.ts` 기준. 하드코딩 금지. 모서리 넉넉히(`rounded-xl`).

## 3. 단계별 작업

### 단계 1 — 서버: support 앱 + 모델 + 마이그레이션
- `python manage.py startapp support`
- `Inquiry` 모델 작성
- `INSTALLED_APPS` 등록
- 마이그레이션

### 단계 2 — 서버: serializer + views + urls
- `InquirySerializer`: POST용(content만), GET용(전체 필드)
- ViewSet or APIView 4개: 등록·목록·unread-count·read
- `support/urls.py` → `config/urls.py`에 `/api/support/` 등록
- 인증·퍼미션 확인

### 단계 3 — 프론트: API 타입 + 훅
- `api/support.ts`: `Inquiry` 타입, `postInquiry` / `getInquiries` / `getUnreadCount` / `patchRead`
- `hooks.ts` 또는 별도 파일에 `useInquiries`, `usePostInquiry`, `useUnreadInquiryCount`, `useMarkInquiryRead`

### 단계 4 — 프론트: InquiryScreen 화면
- `screens/main/InquiryScreen.tsx` 구현 (목록 + 작성 박스)
- `navigation/types.ts`에 `Inquiry: undefined` 추가
- Main 스택에 `InquiryScreen` 등록

### 단계 5 — 프론트: Settings 연결 + Gmail 제거 + 뱃지
- `SettingsScreen`: `openSupport` 제거, `SUPPORT_EMAIL`/`SUPPORT_MAIL` 상수 제거
- ListRow "문의하기" → `navigation.navigate('Inquiry')`
- `useUnreadInquiryCount` 연결 → 미확인 답변 있을 때 ListRow에 dot 뱃지
