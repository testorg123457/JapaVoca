## 단계 1 — Supabase DB 준비

### 사용자 액션
1. [supabase.com](https://supabase.com) → New project 생성
2. **Settings → Database → Connection string** 탭에서  
   **Transaction pooler** 선택 (port **6543**) → URI 복사
   ```
   postgresql://postgres.xxxx:[PASSWORD]@aws-0-ap-northeast-1.pooler.supabase.com:6543/postgres
   ```
3. 복사한 URL을 Claude에게 전달 (아래 `.env.production` 작성 시 사용)

> **왜 Transaction pooler?** Cloud Run은 요청마다 새 연결을 맺을 수 있어 DB 연결 수가 폭발함.  
> Supabase Transaction pooler(pgBouncer)가 연결을 재사용해 해결.

## 단계 2 — Django 프로덕션 설정 (Claude 담당)

### 2-1. 추가 패키지
```
gunicorn        # WSGI 서버 (Cloud Run 표준)
whitenoise      # 정적 파일 서빙 (별도 스토리지 없이)
```
`requirements.txt`에 추가 후 `pip install` 불필요 (Dockerfile에서 처리).

### 2-2. settings.py 수정 포인트
현재 `settings.py`가 환경 변수를 이미 읽는 구조라 프로덕션 전용 파일 없이  
**환경 변수만 올바르게 설정**하면 됨.

추가/변경이 필요한 항목:

| 항목 | 현재 | 프로덕션 |
|------|------|----------|
| `DEBUG` | `True` | `False` (env: `DEBUG=False`) |
| `ALLOWED_HOSTS` | localhost | Cloud Run URL + 커스텀 도메인 |
| DB 설정 | 개별 `DB_*` 변수 | `DATABASE_URL` 단일 변수로 통일 |
| Whitenoise | 없음 | Middleware에 추가, `STATIC_ROOT` 설정 |
| `SECURE_PROXY_SSL_HEADER` | 없음 | Cloud Run 뒤 HTTPS 처리 |
| Celery / Redis | 있음 | **제거 완료** — 쓰는 곳이 없었음 |

> QuizLog 7일 삭제는 이미 management command(`delete_old_quiz_logs`)로 구현돼 있음.  
> 나중에 Cloud Scheduler + Cloud Run Job으로 `python manage.py delete_old_quiz_logs` 주기 실행하면 끝.

### 2-3. `.env.production` 구성 (Claude가 템플릿 생성)
```env
SECRET_KEY=<생성>
DEBUG=False
ALLOWED_HOSTS=<Cloud Run URL>.run.app
DATABASE_URL=postgresql://postgres.xxxx:[PW]@pooler.supabase.com:6543/postgres
CORS_ALLOWED_ORIGINS=https://your-domain.com
GOOGLE_CLIENT_ID=<구글 OAuth 클라이언트 ID>
ADMOB_APP_ID=
ADMOB_REWARDED_UNIT_ID=
ADMOB_SSV_VERIFY=false
GIFTICON_PROVIDER_URL=
GIFTICON_PROVIDER_KEY=
```

## 단계 3 — Dockerfile 작성 (Claude 담당)

```dockerfile
# 예상 구조 (Claude가 실제 작성)
FROM python:3.12-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
RUN python manage.py collectstatic --noinput
EXPOSE 8080
CMD ["gunicorn", "config.wsgi:application", "--bind", "0.0.0.0:8080", "--workers", "2"]
```

## 단계 4 — Cloud Run 배포 (Claude 담당, gcloud 명령어 실행)

### 4-1. 사전 확인 사항 (사용자)
```bash
! gcloud auth login
! gcloud config set project <PROJECT_ID>
# 아래 API 활성화 (console.cloud.google.com → API 라이브러리에서 검색)
# - Cloud Run API
# - Artifact Registry API
# - Cloud Build API
```

### 4-2. 배포 순서 (Claude 실행)
1. Docker 이미지 빌드 & Artifact Registry 푸시
2. Cloud Run 서비스 배포 (환경 변수 주입)
3. `gcloud run exec` 또는 별도 Job으로 `migrate` 실행
4. 배포 URL 확인 → 모바일 `.env`의 `API_BASE_URL` 교체

### 4-3. 배포 명령어 (Claude가 실행할 예정)
```bash
# 이미지 빌드 & 푸시
gcloud builds submit --tag asia-northeast3-docker.pkg.dev/<PROJECT_ID>/japavoca/server

# Cloud Run 배포
gcloud run deploy japavoca-server \
  --image asia-northeast3-docker.pkg.dev/<PROJECT_ID>/japavoca/server \
  --region asia-northeast3 \
  --platform managed \
  --allow-unauthenticated \
  --set-env-vars "SECRET_KEY=...,DATABASE_URL=...,..."

# DB 마이그레이션 (Cloud Run Job으로 1회 실행)
gcloud run jobs create migrate-job \
  --image asia-northeast3-docker.pkg.dev/<PROJECT_ID>/japavoca/server \
  --command python,manage.py,migrate \
  --region asia-northeast3 \
  --set-env-vars "DATABASE_URL=..."
gcloud run jobs execute migrate-job --region asia-northeast3
```
