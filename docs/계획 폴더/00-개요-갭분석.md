## 3. 결론 — 추가/변경 요약
- **DB 변경**: `User`에 `jlpt_level_word`/`jlpt_level_kanji`, `provider`/`kakao_uid`(+`google_uid` nullable), 푸시 환경설정 필드.
- **신규 테이블**: `Notification`, `PushToken`.
- **신규 API**: 출석 월별, 알림 목록/미읽음수/읽음, 푸시토큰 등록/해제, 카카오 로그인.
- **프론트**: 설정 급수 2개·푸시 토글·구매내역·알림화면+헤더 배지·카카오 버튼·출석 달력 실데이터화.

## 4. 자율 구현 vs 선결조건
- **완전 자율 구현(외부 의존 없음)**: A(급수 분리), B(출석 월별), C(인앱 알림), D(구매내역 UI), G(문의 정리).
- **백엔드는 자율 + 클라/크리덴셜 선결**:
  - E 푸시: 백엔드(PushToken·등록 API·설정·발송 추상화) 구현. **실제 FCM 발송은 Firebase 서비스계정 + 클라 `@react-native-firebase` + `google-services.json` 필요** → 발송 provider는 로그 stub, 추후 크리덴셜 주입.
  - F 카카오: 백엔드(provider 리팩 + 카카오 토큰 검증 로그인) 구현. **클라 카카오 SDK + 네이티브 앱키 필요** → 버튼은 키 없으면 안내.
- 선결조건 상세: [03-구현순서-선결조건.md](03-구현순서-선결조건.md).

## 5. 원칙 (CLAUDE.md 준수)
- 캐시 정합성(원장+잔액 트랜잭션) 로직은 **건드리지 않는다** — 알림/푸시는 별도 도메인.
- 마이그레이션은 데이터 백필 포함, 기존 구글 로그인/데이터 무손상.
- 비밀키는 `.env`로 분리, 코드 하드코딩 금지.
- 새 의존성 추가 시 이유 한 줄 기록.
