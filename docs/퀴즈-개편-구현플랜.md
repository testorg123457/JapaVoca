# 퀴즈 개편 — 구현 플랜

> 화면/UX 명세는 [`퀴즈-개편-구조.md`](퀴즈-개편-구조.md). 이 문서는 **어떻게 구현하는가**(데이터 모델·엔드포인트·프론트 구조·의존성·작업 순서)를 다룬다.
> 작업 규칙: [`CLAUDE.md`](../CLAUDE.md) — ⚠️ **캐시 정합성**(원장+잔액 트랜잭션, 상자=서버 권위)은 절대 유지.
> 관련 코드: 백엔드 `apps/server/learning/`, `apps/server/rewards/`, `apps/server/content/` · 프론트 `apps/mobile/src/screens/quiz/`, `apps/mobile/src/api/`.

## 0. 핵심 설계 결정 (먼저 읽기)

| # | 결정 | 근거 |
|---|---|---|
| D1 | **세트는 서버가 발급·관리** (`QuizSet` 모델). 상자 3개 캡·1시간 쿨다운을 서버에서 강제. | 상자=돈 → 캡은 반드시 서버 권위(CLAUDE.md). |
| D2 | **배치 응답에 정답키 + 해설데이터 포함**(오프라인 즉시 채점/해설용). | 오프라인 즉시 피드백 요구. 아래 "보안" 참고. |
| D3 | **상자 지급은 온라인 토큰 채점 경로에서만.** 서버는 클라의 is_correct를 절대 신뢰 안 함. | 정답키 노출이 보상 추출을 늘리지 못하게(캡이 한계). |
| D4 | **음성 = `react-native-tts`** (기기 일본어 음성). 오디오 파일 파이프라인 없음. | 9천+ 항목 음성파일 생성/호스팅 비용 회피. MVP 적합. |
| D5 | **북마크 = 서버 모델 + 오프라인 낙관적 토글**(MMKV 큐 → 동기화). | 오프라인 토글 요구. |

### 보안 메모 (D2/D3)
정답키를 배치에 포함하면 온라인에서도 클라가 정답을 미리 안다. 하지만 **이 앱의 보상 한계는 "정답 난이도"가 아니라 "상자 캡(세트 3 / 일일 50) + answer_ms + Play Integrity"** 이다. 정답을 알아도 캡 이상 상자를 못 얻으므로 경제가 깨지지 않는다. 서버는 여전히 **서명 토큰으로 독립 채점**해 상자를 확정하고, 클라가 보낸 정오답을 신뢰하지 않는다. → 정답키 노출은 의도된 트레이드오프.
- 더 보수적인 대안(정답키 미포함 + 오프라인은 답안 큐잉 후 온라인 시 채점·해설 지연)도 가능하나, "오프라인 즉시 해설" 요구를 못 맞춘다. 본 플랜은 즉시 해설을 채택.

## 1. 백엔드 — 출제 세트 모델

### 1-1. 새 모델 (`learning/models.py`)

```python
class QuizSet(models.Model):
    """10문제 세트. 상자 3개 캡·1시간 쿨다운의 단위."""
    user = FK(User, related_name='quiz_sets')
    started_at = DateTimeField(auto_now_add=True)
    completed_at = DateTimeField(null=True, blank=True)
    boxes_earned = PositiveIntegerField(default=0)
    answered_count = PositiveIntegerField(default=0)
    # db_table = 'tbl_learning_quizset'

class QuizSetItem(models.Model):
    """세트 내 문항. 재로드 시 재생성 없이 DB에서 직렬화하기 위해 표시용 필드도 저장."""
    quiz_set = FK(QuizSet, related_name='items')
    order = PositiveIntegerField()                        # 1..10
    item_type = CharField(ItemType.choices)               # 'word' | 'kanji' | 'kana'
    word_type = CharField(null=True, blank=True)          # 'kanji' | 'kana' | null
    item_id = PositiveIntegerField()
    question_type = CharField(...)
    correct_index = PositiveIntegerField()
    prompt = CharField(max_length=200, blank=True)        # 생성 시 확정·저장
    reading = CharField(max_length=100, blank=True)       # 생성 시 확정·저장
    jlpt_level = CharField(max_length=2, blank=True)      # 생성 시 확정·저장
    choices_json = JSONField(default=list)                 # [{index, text}, ...] 셔플 완료 순서
    answered = BooleanField(default=False)
    is_correct = BooleanField(null=True)
    # db_table = 'tbl_learning_quizsetitem'
    # unique_together = (quiz_set, order)
```

> `prompt`/`reading`/`jlpt_level`/`choices_json` 저장 이유: 세트 재로드(`_serialize_existing_set`) 시 content 테이블 재조회 없이 그대로 직렬화. `choices_json`은 셔플 순서가 랜덤이라 반드시 저장해야 `correct_index`가 유효함.
> `word_type` 이유: `item_type='word'`는 kanji_word/kana_word 모두 포함. 재로드·detail 분기·SRS 트랙 오염 방지에 필요.
> `detail`은 저장하지 않음 — content 모델에서 결정론적으로 재파생 가능(`_item_detail`).

- `SET_SIZE = 10`, `SET_BOX_CAP = 3`, `SET_COOLDOWN = timedelta(hours=1)` 상수.

### 1-3. 세트 생성 로직 (`learning/services.py`)
- `build_quiz_set(user)`:
  1. 활성 세트(미완료) 있으면 그대로 반환.
  2. 마지막 완료 세트의 `completed_at + SET_COOLDOWN > now` → 쿨다운 응답.
  3. 아니면 `build_question(user)`을 **10회 반복**해 항목 선정(`resolve_study`/`_pick_item_id`/`_distractor_texts` 재사용) → `QuizSet`+`QuizSetItem` 생성. 같은 세트 내 **중복 item 회피**(이미 뽑은 id 제외).
  4. 각 문항의 `correct_index`는 DB(QuizSetItem)에 저장, 토큰에도 인코딩(`set_id`/`order` 포함).

### 1-4. 채점 변경 (`grade_answer`)
- 토큰 payload에 `set_id`, `order` 추가. 채점 시:
  1. 기존 서명/만료/중복 검증 유지(`token_hash`).
  2. `QuizSetItem` 잠금(`select_for_update`) → 이미 `answered`면 거부.
  3. SRS(SM-2) 갱신, QuizLog 기록(현행 유지).
  4. **상자 지급 조건**: `is_correct AND quiz_set.boxes_earned < SET_BOX_CAP AND daily.boxes_earned < MAX_BOXES_PER_DAY`. → 충족 시 상자 생성 + `quiz_set.boxes_earned += 1`. (온라인 여부 검증은 이 경로 자체가 온라인 채점이므로 별도 불필요.)
  5. `quiz_set.answered_count += 1`; 10 도달 시 `completed_at = now`(쿨다운 시작).
  6. 응답에 `set_boxes_earned`(현재 세트 누적 상자 수), `set_completed`(세트 완료 여부) 포함.
- **`answer_ms` 오프라인 큐 처리**: 오프라인에서는 선택 시점 타임스탬프(`answered_at`)를 `pendingAnswers` 항목에 저장. sync 시 서버는 `answered_at` 기준으로 `answer_ms`를 검증하고 지나치게 빠른 응답(어뷰징)만 필터링.
- **오프라인 동기화 채점**: 별도 경로(`POST /api/quiz/sync/`, 1-5) — SRS/통계만 갱신, **상자 절대 미지급**.

### 1-5. 오프라인 동기화 엔드포인트
| `POST /api/quiz/sync/` | 오프라인에서 푼 답안 배열 `[{question_token, choice_index, answer_ms, answered_at}]`을 받아 SRS/QuizLog만 갱신. 상자 미지급. 멱등(`token_hash`). |

- 토큰은 서명되어 있으므로 서버가 정오답을 독립 판정(클라 is_correct 불신). 단 **상자 로직은 스킵**.
- **`answered_count` / `completed_at` 갱신**: 동기화 채점 시에도 `QuizSetItem.answered = True`, `quiz_set.answered_count += 1`을 갱신한다. 10 도달 시 `completed_at = answered_at`(마지막 답안 시각 기준)으로 설정해 쿨다운을 시작.
- **토큰 만료 처리**: 오프라인 큐에 쌓인 토큰은 TTL이 지날 수 있다. sync 엔드포인트는 TTL 만료 토큰도 `set_id` + 세트 생존 여부로 재검증(1-6절 방식). 세트 자체가 만료된 항목은 SRS 갱신을 건너뛰고 무시(부분 동기화 허용, 상자는 어차피 미지급).

### 1-6. 토큰 TTL
- 현재 `QUESTION_TTL_SECONDS=300`(5분)은 세트(최대 1시간+) 동안 만료됨.
- **채택 방식**: `signing.loads()`의 `max_age` 검사를 세트 경로에서만 **우회**하고, 대신 payload의 `set_id`로 `QuizSet` 생존 여부를 확인해 유효성 판정. 서명 위조 차단은 여전히 동작. `QUESTION_TTL_SECONDS`(5분)는 기존 `/api/quiz/next/` 단발 경로에만 적용 유지.
  ```python
  # grade_answer / sync: set_id 포함 토큰이면 TTL 우회 + 세트 생존 검증
  payload = signing.loads(token, salt=QUESTION_SALT, max_age=None)  # 서명만 검증
  if 'sid' in payload:
      quiz_set = QuizSet.objects.filter(id=payload['sid'], user=user).first()
      if not quiz_set:
          raise InvalidQuestionToken('세트가 존재하지 않습니다.')
  else:
      signing.loads(token, salt=QUESTION_SALT, max_age=QUESTION_TTL_SECONDS)  # 기존 TTL 검증
  ```
- 멱등성은 `token_hash`로 유지.

## 3. 백엔드 — 해설 detail payload

- `build_quiz_set`에서 각 항목의 `detail`을 채운다. 기존 `_item_texts`/`_item_extra` 확장:
  - **한자**: `components`, `stroke_count`, `on_reading`, `kun_reading`, `meaning_ko`. (`Kanji` 모델에 다 있음.)
  - **한자단어**: `surface`, `reading`, `meaning_ko`, `jlpt_level`. 구성자는 **단어 표기 내 한자를 `Kanji`에서 조회해 매핑**(가능한 글자만). 미보유 시 생략.
  - **가나단어**: `surface`, `reading`, `meaning_ko`.
  - **가나**: `character`, `romaji`, `script`.
- ⚠️ **부수 전용 필드 없음** → `components`(구성자)로 표기. 향후 `Kanji`에 `radical` 컬럼 추가 검토(별도 작업).

## 4. 프론트 — 데이터 레이어 (오프라인)

### 4-1. 의존성 추가
| 패키지 | 상태 | 이유 |
|---|---|---|
| `react-native-tts` | **미설치** (별도 단계) | 한자·가나·단어 발음. 네이티브 링크 필요 → `run-android` 재빌드 필요. |
| `@react-native-community/netinfo` | **미설치** (별도 단계) | 온/오프라인 전환 이벤트 구독. 현재는 API 오류 catch로 오프라인 감지 대체. |

> MMKV·react-query·reanimated·svg·lottie는 이미 설치됨(재사용). 의존성 추가 시 `package.json` + 한 줄 사유 기록(CLAUDE.md).

### 4-2. 세트 캐시 (MMKV) — `store/quizSet.ts`
- 세트 전체(`quiz.activeSet`), 커서(`quiz.cursor`), 오프라인 큐(`quiz.pendingAnswers`)를 MMKV에 저장.
- 온라인 진입 시: 서버에서 세트 수신 → MMKV 덮어쓰기. 오프라인 진입 시(API 오류 발생): MMKV 캐시 세트 사용.
- 커서는 로컬 단순 유지. 다른 기기 동기화 or 서버 answered_count 기준 보정은 향후 과제.
- 오프라인에서 푼 답안은 `quiz.pendingAnswers` 큐에 적재. 항목: `{ question_token, choice_index, answer_ms?, answered_at: ISO string }`.
- 온라인 복귀 시(현재는 `loadSet()` 재호출 시 pending flush 실행): `POST /api/quiz/sync/`로 flush.

### 4-3. 온/오프라인 감지 방식
- **현재 구현**: `getQuizSet()` 호출 시 `err.response`가 없으면(네트워크 오류) 오프라인으로 판단, MMKV 캐시로 전환. 채점 실패 시도 동일 분기.
- **향후**: `@react-native-community/netinfo` 설치 후 이벤트 기반 감지로 교체. 온라인 전환 시 pending flush + react-query invalidate.

### 4-4. 채점 분기
```
사용자가 선택지 탭
 ├─ 온라인  → POST /api/quiz/answer/  (서버 채점 + 상자)
 │             ├─ 성공: is_correct·box_id·set_boxes_earned·set_completed 확정 → reveal
 │             └─ 실패(err.response 없음): 로컬 채점(answer_index) + pendingAnswers 큐 → reveal(상자 없음)
 └─ 오프라인 → 로컬 채점(answer_index) → reveal(상자 없음) → pendingAnswers 큐 적재
```
- 해설 detail은 배치 세트에 이미 포함(`QuizSetQuestion.detail`) — 서버 재요청 없음.
- 온라인 채점 실패 폴백: sync 시 `already_graded` 응답은 정상 처리(멱등).

### 5-1. 상태기계 (구현됨)

```typescript
type Phase =
  | { type: 'loading' }
  | { type: 'playing'; cursor: number; set: QuizSetResponse }
  | { type: 'reveal'; cursor: number; set: QuizSetResponse;
      isCorrect: boolean; boxGrade: BoxGrade | null; offlineMode: boolean }
  | { type: 'cooldown'; cooldownUntil: string }
  | { type: 'noContent' }
```
- `loading`: 세트 로드 중. 네트워크 오류 시 MMKV 캐시로 전환. 캐시도 없으면 `noContent`.
- `playing`: 문제·음성버튼·선택지. 선택 시 채점(4-4) 후 `reveal`.
- `reveal`: 해설 패널(결과 헤더 + detail + 북마크/음성 + "다음 문제"). **자동 타이머 없음**.
- `cooldown`: 쿨다운 카운트다운. `cooldownUntil` 만료 시 **자동으로 `loadSet()` 재호출** → `loading`으로 전환. `setTimeout` 기반.
- `noContent`: 풀 문제 없음 정적 화면.

### 5-2. 컴포넌트 구조 (LockQuizScreen.tsx 내 inline)
별도 파일 분리 없이 `LockQuizScreen.tsx` 안에 함수 컴포넌트로 정의.

## 8. 미해결 / 추후 결정

- **부수(radical) 데이터**: 전용 필드 없음. MVP는 `components`로 대체. 정식 부수 표기가 필요하면 `Kanji.radical` 컬럼 + 시드 보강(별도).
- **한자단어 구성자 분해**: 단어→구성 한자 매핑 데이터 미보유. 런타임 `Kanji` 조회로 부분 제공, 한계 존재.
- **TTS 품질/가용성**: 기기에 일본어 음성 없으면 비활성. 품질 이슈 크면 향후 서버 오디오 파일(프리렌더)로 승급 검토.
- **세트 쿨다운 UX**: 1시간이 길게 느껴지면 "광고 보고 즉시 새 세트"(보상형 광고 연계) 옵션 고려 — 수익모델과 맞음(별도).
- **세트 생성 중 콘텐츠 부족**: `build_quiz_set` 중 10개를 채우기 전 `NoContent` 발생 시 처리 방식 미결(부분 세트 허용 vs 세트 자체 미생성). MVP에서는 최소 5개 확보 가능하면 부분 세트 허용, 5개 미만이면 NoContent 화면 표시 방향으로 구현하되, 정책 확정은 백엔드 2단계 시작 전에 결정.
- **`pendingAnswers` 큐 토큰 만료**: 1-6절 방식으로 sync 엔드포인트에서 세트 생존 기준으로 재검증하므로 처리 가능. 단, 세트 자체가 사라진(서버에서 purge 등) 극단적 경우에는 큐 항목 폐기 처리(에러 무시 후 큐 제거).
- **기존 `/api/quiz/next/`**: `LockQuizScreen`이 세트 API로 완전 전환됐으므로 제거 후보. 다른 화면에서 사용 여부 확인 후 결정.
- **북마크 item_id**: `AnswerReveal`에서 `item_id: 0`으로 임시 처리 중. `QuizSetQuestion`에 `item_id` 노출 필요(6절 7단계).
- **netinfo 미설치**: 현재 API 오류 catch 방식. 온라인 복귀 시 pending 자동 flush 없음 — `loadSet()` 재호출 시에만 flush됨(6절 9단계에서 해결).
