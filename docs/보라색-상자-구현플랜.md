# 보라색 상자(레어 상자) 구현 플랜

> 상자 개봉에 **희귀도 연출**을 도입한다. 정답 시 굴리는 상자 등급을 **일반 / 보라(레어) 2종**으로 단순화하고,
> 보라 상자는 낮은 확률(10%)로 등장하며 일반보다 **높은 캐시**를 준다.
> 일반 상자 보상은 현재의 **1/10 수준**으로 낮추고, 보라 상자는 그 최대치 이상을 지급한다.

작업 규칙은 루트 [`CLAUDE.md`](../CLAUDE.md), 현황은 [`프로잭트-현황.md`](프로잭트-현황.md) 참고.
**⚠️ 캐시 정합성 원칙**(Wallet+Ledger 원자 트랜잭션)은 이번 작업에서 그대로 유지된다. 지급 경로(`open_cash_box → earn`)는 바꾸지 않고 **금액 범위와 등급 롤만** 바꾼다.

---

## 1. 목표 / 결정 사항

- **등급 2종화**: 기존 5단계(`normal/rare/epic/legendary/jackpot`)를 **`normal`(일반) / `purple`(보라)** 2종으로 단순화. rare~jackpot 등급 롤은 폐기(생성 안 함).
- **등장 확률**: 정답 시 상자 생성 롤 = **일반 90% / 보라 10%**.
- **보상 캐시** (⚠️ 임시값 — 나중에 조정):
  - 일반: 현재 `(10, 40)` → **1/10 & 최대치 축소** → `(1, 4)`
  - 보라: **일반 최대치 초과부터** → `(5, 15)` (예시. "21~50" 같은 느낌으로 폭넓게 줄 수 있음)
- **연출 분기**: 개봉 시 등급에 따라 **다른 Lottie 애니메이션** 재생.
  - 일반 → 기존 `gift-box-animation.json`
  - 보라 → 루트의 `Present for you보라색.json` (mobile 에셋으로 이동)
- **정합성 불변**: 지급은 여전히 서버가 확정(`open_cash_box`), 클라 금액 불신뢰. 원장/잔액 트랜잭션 그대로.

> 참고: 프론트는 이미 개봉 응답에서 `grade`를 받고 있으나(현재 미사용), 상자를 전부 동일하게 보여준다.
> 이번 작업으로 `grade`를 실제 연출 분기에 사용한다.

## 2. 백엔드 변경 (apps/server)

### 2.1 `rewards/models.py` — `CashBox.Grade`
- `Grade` 초이스를 **`NORMAL` / `PURPLE`** 2종으로 축소.
  - `RARE/EPIC/LEGENDARY/JACKPOT` 제거.
- **마이그레이션 주의**: `grade`는 `CharField(choices=...)`라 **DB 컬럼 타입·스키마는 안 바뀐다**(varchar(10) 그대로). choices 변경은 Django `AlterField` 마이그레이션 1건을 만들지만 SQL 레벨은 사실상 no-op. 그래도 `migrate`는 돌려야 함.
  - **기존 데이터(⚠️ 실측 — 반드시 필요)**: 현재 개발 DB 미개봉 상자 분포 = **`normal` 10 / `rare` 2 / `epic` 3**.
    등급을 2종으로 줄이면 이 **rare 2 + epic 3 = 5개**를 개봉할 때 `BOX_REWARD_RANGE[grade]`에 키가 없어 **KeyError**가 난다.
    → **데이터 마이그레이션**(같은 마이그레이션 파일 안에서 `RunPython`)으로 미개봉 상자의 옛 등급(rare/epic/legendary/jackpot)을 `normal`로 일괄 치환한다. 개봉 완료된 상자는 이미 `reward_cash` 확정이라 안 건드림.

### 2.2 `rewards/services.py` — 보상 범위
```python
BOX_REWARD_RANGE = {
    CashBox.Grade.NORMAL: (1, 4),    # 기존 (10,40)의 1/10, 최대치 축소 (임시)
    CashBox.Grade.PURPLE: (5, 15),   # 일반 최대치(4) 초과부터 (임시)
}
```
- `open_cash_box()` 로직은 **변경 없음** — `BOX_REWARD_RANGE[box.grade]`로 그대로 동작.
- 값은 상수 한 곳(딕셔너리)만 고치면 조정되도록 유지.

### 2.3 `learning/services.py` — 등급 롤
```python
# 정답 시 상자 등급 가중치 (일반 90% / 보라 10%)
_BOX_GRADE_WEIGHTS = [
    (CashBox.Grade.NORMAL, 90),
    (CashBox.Grade.PURPLE, 10),
]
```
- `_roll_box_grade()` 함수 시그니처/호출부(`CashBox.objects.create(user=user, grade=_roll_box_grade())`)는 그대로.

### 2.4 확인 사항
- `rewards/serializers.py` / `views.py`(`OpenBoxView`)가 `grade`를 응답에 그대로 실어보내는지 확인(현재 계약: `{ box_id, grade, reward_cash, balance_after }`). 등급 값만 `normal|purple`로 바뀜.
- admin/테스트에서 옛 등급 문자열 하드코딩 참조 없는지 grep.

## 3. 프론트 변경 (apps/mobile/src)

### 3.1 에셋 이동
- `Present for you보라색.json`(루트, 33KB) → `apps/mobile/src/assets/purple-box-animation.json`으로 이동/리네임.
  - (파일명 공백·한글 회피 위해 영문 리네임 권장.)

### 3.2 타입 (`api/hooks.ts`, `api/boxes.ts`)
- `BoxGrade = 'normal' | 'purple'` 로 축소.

### 3.3 `screens/quiz/BoxOpenScreen.tsx` — 연출 분기
- 등급별 Lottie source 매핑:
  ```ts
  const boxAnim = box.grade === 'purple'
    ? require('../../assets/purple-box-animation.json')
    : require('../../assets/gift-box-animation.json');
  ```
  - ⚠️ `require`는 정적이어야 하므로 삼항으로 미리 두 source를 분기.
- 현재 개봉 애니는 **`box.grade`(생성 시 확정된 등급)** 기준으로 고름. 개봉 응답(`result.grade`)과 항상 일치.
  - 단, 현재 Lottie는 `phase==='sealed'`부터 화면에 떠 있음(개봉 전에도 보임). 개봉 전부터 보라 상자를 노출하면 서프라이즈가 줄어듦
    → **결정 필요(§5)**: 개봉 전엔 공통 밀봉 상태로 두고 개봉 순간에만 등급별로 바꿀지, 인벤토리부터 보라를 보여줄지.
- 보라 개봉 시 보상 텍스트/글로우 색을 보라 계열로 살짝 강조(선택, 디자인 토큰 사용).

### 3.4 홈 인벤토리 (`screens/main/HomeScreen.tsx`)
- 미개봉 상자 목록(`useBoxes`)에서 보라 상자를 다르게 표시할지 여부 → §5 결정에 따름.
  - 서프라이즈 유지하려면 인벤토리에선 **등급 숨김**(전부 동일 아이콘), 개봉 순간에만 공개 권장.

## 4. 테스트 / 검증

- **백엔드 단위테스트**(`rewards/tests.py`, `learning/tests.py`):
  - `_roll_box_grade()`가 `normal|purple`만 반환.
  - 보라/일반 각각 개봉 시 `reward_cash`가 지정 범위 내.
  - `open_cash_box` 후 `wallet_is_consistent(user)` True(정합성 불변식 유지).
  - 옛 등급 상자 데이터 마이그레이션 후 개봉 시 KeyError 안 남.
- **수동 플로우**: 정답 반복 → 상자 여러 개 확보 → 개봉하며 보라 등장/연출 확인(10%라 표본 필요. 필요 시 임시로 가중치 올려 확인 후 원복).

## 5. 열린 결정 (착수 전 합의)

1. **캐시 범위 확정값**: 일반 `(1,4)` / 보라 `(5,15)` 가 임시. 실제 값 알려주면 반영(상수 한 곳 수정).
2. **서프라이즈 정책**: 보라 상자를 (a) 개봉 순간에만 공개 / (b) 인벤토리·밀봉 상태부터 보라로 노출. → 연출 위치가 달라짐.
3. **옛 등급 데이터**: 개발 DB라 그냥 미개봉분 `normal` 치환으로 충분한지(운영 데이터 없다고 가정).

## 6. 작업 순서 (단계별)

1. 백엔드 모델 등급 축소 + 마이그레이션(+ 데이터 마이그레이션) → 2. 보상 범위/롤 상수 수정 → 3. 서버 테스트.
4. 애셋 이동/리네임 → 5. 프론트 타입 축소 → 6. `BoxOpenScreen` 연출 분기 → 7. 수동 플로우 검증.

각 단계 끝에 한 일/막힌 점/다음 단계 보고(§CLAUDE.md 작업 습관).
