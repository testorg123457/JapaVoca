# 일본어 한자 보카 — 개발 계획서 (초안)

> **서비스명**: 일본어 한자 보카 (영문: **JapaVoca**)
> **스토어 부제(노출 키워드)**: JLPT 잠금화면 일본어 단어 퀴즈, 돈버는 앱, 출석 현금 리워드, 앱테크
>
> 이 문서는 프로젝트 작업 시 참고할 **개발 지침 / 기획 베이스**입니다.
> "확정"으로 표기되지 않은 수치·구조는 초안이며 개발 중 조정 가능합니다.

## 0. 프로젝트 개요

일본어(단어·한자) 학습 + 리워드(캐시) 앱.
사용자가 퀴즈를 맞히거나 출석하면 캐시를 지급하고, 캐시는 기프티콘/상품권으로 교환.
수익 모델은 **캐시워크형** — 캐시 수령/교환 시 Google AdMob 보상형 광고 시청.

- **플랫폼**: Android (1차)
- **타겟 사용자**
  - 일반 일본어 학습자
  - JLPT(N1~N5) 취득 목표자
- **핵심 루프**: 퀴즈 풀기 → 캐시상자 획득 → (광고 시청) → 캐시 적립 → 기프티콘 교환(광고 시청)

### 확정된 의사결정 요약
| 항목 | 결정 |
|------|------|
| 클라이언트 | React Native |
| 로그인 | 구글 소셜 로그인 |
| 기프티콘 | B2B 기프티콘 API 연동 (사업자등록 보유) |
| 캐시 환율 방향 | 방향 A — 환율을 짜게, 화면상 큰 숫자 유지 |
| 학습 모드 | 단어 퀴즈 / 한자 퀴즈 **분리** |
| 출제 알고리즘 | SRS(간격 반복) 도입 |
| 학습 범위 | 사용자가 JLPT 급수(N5~N1) 선택 |
| 오답 생성 | 동일 품사·동일 급수의 유사 항목에서 추출 |
| 캐시상자 | 문제 정답 시마다 1상자 획득(인벤토리 적립), 상자 3개당 광고 1회 |
| 보너스 | 출석 보너스 + 연속 출석 보너스 둘 다 포함 |
| DB | PostgreSQL (캐시 정합성 우선) |
| 캐시 내역 | 통합 원장 `ledger` + `direction`(earn/use)로 화면 구분 |
| UI/디자인 | gluestack-ui + NativeWind, 토스st 커스텀 테마, Pretendard 폰트 |
| 잠금화면 학습 | 부제에 명시 — 안드로이드 잠금화면 위젯/오버레이로 단어 노출·간단 퀴즈(별도 구현 필요, 우선순위 검토) |

## 1. 기능 플로우

### 1.1 전체 사용자 여정
```
[앱 실행]
  → 최초: 구글 로그인 → 학습 급수 선택(N5~N1) → 모드 선택(단어/한자)
  → 재방문: 홈
[홈]
  → 출석 체크(보너스 캐시) · 오늘의 학습 진행률 · 보유 캐시 · 상자 인벤토리
[학습(퀴즈)]
  → 4지선다 문제 출제 → 정답 시 캐시상자 획득(인벤토리)
  → 세션 종료 후 상자 개봉 단계로
[상자 개봉]
  → 상자 3개당 보상형 광고 1회 → 등급별 캐시 지급(연출)
[지갑/교환소]
  → 보유 캐시로 기프티콘 교환 → 교환 시 보상형 광고 시청 → 발송
```

### 1.2 화면 목록 (1차 MVP)
- 로그인 / 온보딩(급수 선택)
- 홈(대시보드): 출석, 진행률, 캐시 잔액, 상자 개수
- 학습 모드 선택(단어 / 한자)
- 퀴즈 화면(4지선다)
- 상자 개봉 화면(광고 연동)
- 지갑 / 기프티콘 교환소
- 교환 내역 / 캐시 적립 내역
- 설정(급수 변경, 로그아웃, 알림)

### 1.3 광고 노출 지점 (AdMob 보상형)
| 지점 | 빈도 | 비고 |
|------|------|------|
| 상자 개봉 | 상자 3개당 광고 1회 | **상자 획득 ≠ 광고**. 인벤토리에 쌓아두고 개봉 시점에만 광고 |
| 기프티콘 교환 | 교환 1건당 1회 | 캐시워크 모델 핵심 |
| (선택) 추가 보너스 상자 | 1일 N회 한도 | "광고 보고 보너스 상자 1개 더" 유도 |

> ⚠️ **AdMob 정책 주의**: 보상형 광고를 너무 촘촘히(특히 인터스티셜과 혼용) 띄우면 노출 제한·정책 위반 위험. 광고 사이 최소 간격 및 1일 노출 상한을 두는 게 안전.

## 2. 학습 방식 및 UX 플로우

### 2.1 퀴즈 형식 (1차)
- **4지선다** 2종
  - 단어 → 뜻
  - 뜻 → 단어
- **오답(distractor) 생성 규칙**: 정답과 **같은 품사 + 같은 JLPT 급수**의 다른 단어/한자에서 추출 → 난이도와 변별력 확보
- 한자 모드도 동일 구조(한자→뜻/음훈독, 뜻→한자 등)

> 향후 확장 후보(2차): 입력형(히라가나 타이핑), 듣기(TTS 음성→단어), 한자 획순 등

### 2.2 학습 모드 분리
- **단어 퀴즈 모드** / **한자 퀴즈 모드**를 별도 진입점으로 분리
- 각 모드는 사용자가 선택한 급수 범위 내에서 출제

### 2.3 급수 선택
- 온보딩 시 N5~N1 중 선택, 설정에서 변경 가능
- (선택 확장) "현재 급수 + 한 단계 아래"를 섞어 복습 강화

### 2.4 출제 알고리즘 — SRS(간격 반복)
틀린 단어를 다시 출제하는 로직을 어차피 만들어야 하므로, 처음부터 SRS로 설계해 암기 효과 극대화.

- 각 (사용자 × 단어/한자) 쌍마다 학습 상태 보관
- 간소화된 SM-2 계열 알고리즘 사용 권장:
  - 정답 시: 다음 복습 간격(interval) 증가
  - 오답 시: 간격 초기화, 단기 재출제 큐로
- 출제 우선순위: **복습 기한 도래 항목 > 신규 항목 > 여유 항목**
- 세션 구성 예: 복습 70% + 신규 30%

#### SRS 상태 전이(예시)
```
신규 → (정답) → 간격 1일 → (정답) → 3일 → 7일 → 16일 ...
        (오답) → 간격 리셋 + 단기 재출제
```

### 2.5 게임화(캐시상자) UX
- 정답마다 상자 1개 획득 → 인벤토리 적립(이때는 광고 없음)
- 학습 세션 종료 후 "상자 개봉" 단계 진입
- 상자 3개 개봉할 때마다 보상형 광고 1회 → 등급별 캐시 지급 연출(룰렛/개봉 애니메이션)

## 3. 캐시 지급 정책 (확정 방향 A 기준)

### 3.1 환율 (확정 방향, 수치는 조정 가능)
- 사용자 입력 기준: 화면상 **큰 숫자 유지**, 환율은 짜게.
- **권장 환율: 약 12,000캐시 = 100원** (1캐시 ≈ 0.0083원)
  - 근거: 보상형 광고 1회 수익(한국 Android eCPM 기준 약 7~15원) 대비 지급 캐시가 작아야 흑자.
  - 광고 1회당 지급액이 약 1.5~3원 수준이 되도록 상자 기댓값과 함께 튜닝.

> 사용자가 처음 제시한 "1,200캐시=100원(광고 1회당 약 18.75원 지급)"은 **광고 수익보다 지급액이 커서 적자** 구조라 환율을 한 자릿수~두 자릿수 배 짜게 조정함. 화면에 보이는 상자 캐시(60~90 등)는 그대로 둬도 됨 — 환율만 조정하면 적자 해소.

### 3.2 상자 등급 / 확률 테이블 (초안)
상자 3개당 광고 1회. 상자 1개 기대값 약 100캐시 → 광고 1회당 약 300캐시 지급.
권장 환율(12,000캐시=100원) 적용 시 광고 1회당 약 2.5원 지급 → 흑자 여력 확보.

| 등급 | 지급 캐시 | 확률 |
|------|-----------|------|
| 일반 | 60~90 | 80% |
| 레어 | 100~200 | 17% |
| 잭팟 | 400~700 | 3% |

- 기댓값 ≈ 약 100캐시/상자 (확정 전 시뮬레이션 권장)
- 확률·구간은 운영 중 A/B로 조정

### 3.3 하루 최대 지급량 제한 — **둘 것(권장)**
- **일일 캐시 획득 상한** 설정 (예: 무광고 학습으로 받을 수 있는 상자 수 상한 + 보너스 상자 1일 N회)
- 목적: 비용 통제 + 과도한 어뷰징 방지 + 광고 인벤토리 보호
- 권장 구조:
  - 학습 정답 상자: 1일 최대 X개까지 적립 (예: 30개)
  - 광고 보너스 상자: 1일 최대 Y회 (예: 5회)
  - 상한 도달 시 "내일 다시" 안내(리텐션 유도)

### 3.4 레벨/난이도별 차등 지급 — **소폭 차등(권장)**
- 상위 급수(N2/N1)일수록 상자 기댓값 가중치 약간↑ (예: 급수별 1.0 / 1.1 / 1.2 …)
- 단, 차등을 너무 키우면 "쉬운 문제 반복 어뷰징"의 반대급부로 "고급수 어뷰징"이 생길 수 있어 소폭으로.

### 3.5 출석 / 연속 출석 보너스 — **포함(확정)**
- **기본 출석**: 1일 1회 출석 시 소액 캐시 또는 상자 1개
- **연속 출석**: 연속 일수에 따라 보너스 증가, 일정 주기로 리셋
  - 예: 7일 누적 사이클
    | 연속일 | 보너스 |
    |--------|--------|
    | 1일 | 기본 |
    | 2~6일 | 점증 |
    | 7일 | 대형 보너스 상자(잭팟 확률↑) |
  - 7일 후 사이클 반복, 하루 거르면 카운트 리셋(또는 "스트릭 보호권"을 광고 시청으로 1회 제공 — 리텐션 장치)
- 출석 보너스 수령에도 광고를 붙일지 여부는 정책 선택(붙이면 수익↑, UX 약간↓)

### 3.6 어뷰징 방지 정책
- **계정 단위 통제**: 구글 로그인 필수 → 1인 1계정 기준선
- **서버 권위 검증**: 캐시 지급/정답 판정은 **반드시 서버에서** 검증(클라 신뢰 금지)
- **광고 보상 검증**: AdMob **SSV(Server-Side Verification)** 사용 → 광고 실제 시청 콜백을 서버가 검증한 뒤 캐시 지급
- **속도/패턴 탐지**:
  - 비정상적으로 빠른 정답 속도(자동 클릭) 탐지 → 쿨다운/플래그
  - 동일 문제 연속 정답 반복 패턴 탐지
  - 비정상 디바이스/에뮬레이터 탐지(Play Integrity API)
- **일일 상한**(3.3)으로 1차 방어
- **이상 행위 대응**: 자동 플래그 → 캐시 적립 보류 → 검토 후 교환 차단/계정 제한
- **교환 단계 검증**: 기프티콘 교환 시 본인 식별(휴대폰/이메일) + 누적 이상 이력 점검

## 4. 기술 스택

### 4.1 권장 구성
| 레이어 | 권장 | 비고 |
|--------|------|------|
| 클라이언트 | **React Native** | 보유 역량 적합, AdMob/소셜로그인 라이브러리 풍부, Android 우선이지만 추후 iOS 확장 용이 |
| UI / 디자인 시스템 | **gluestack-ui + NativeWind(Tailwind)** | 토큰 기반이라 룩을 자유롭게 커스텀(토스st 지향). 검증된 컴포넌트 위에 자체 테마. 폰트는 **Pretendard** |
| 애니메이션/게임화 연출 | **react-native-reanimated + lottie-react-native** | 캐시상자 개봉·잭팟 연출 등. 디자인 시스템이 기본 제공하지 않는 부분 |
| 광고 | **Google AdMob** (보상형 중심) | `react-native-google-mobile-ads` 사용. SSV 필수 |
| 인증 | **Google 소셜 로그인** | Firebase Authentication 활용 권장 |
| 백엔드 | **Django (Python) + DRF** | 보유 역량 + 데이터 정제(파싱/AI 태깅)가 Python 친화적이라 시너지. (Node.js도 가능하나 데이터 파이프라인 때문에 Python 우위) |
| DB | **PostgreSQL** | 트랜잭션 안정성(캐시 잔액 정합성), JSON 컬럼 지원 |
| 캐시/세션 | Redis (선택) | 일일 상한 카운터, 레이트리밋, SRS 큐 캐싱 |
| 푸시/분석 | Firebase (FCM, Analytics) | 출석 리마인드, 리텐션 분석 |
| 인프라 | 단일 VM/컨테이너로 시작 → 필요 시 확장 | 초기 비용 최소화 |

> **왜 React Native + Django 조합인가**
> - 클라이언트는 다룰 수 있는 기술 중 보상형 광고/소셜로그인 생태계가 가장 성숙.
> - 백엔드를 Django로 두면 보유 데이터의 정제(품사 HTML 파싱, 뜻 형식 통일, JLPT AI 태깅)를 같은 언어/환경에서 처리해 파이프라인이 단순해짐.

### 4.x UI / 디자인 방향

> **결정**: 한국 기업이 공개한 "RN용 완성형 디자인 시스템"은 사실상 없음(토스 TDS는 비공개, 당근 Seed Design은 웹 React 중심). 따라서 **토큰 기반 시스템(gluestack-ui) + NativeWind**로 검증된 컴포넌트 위에 **자체 테마**를 입혀 "토스st" 룩을 구현. (Tamagui도 후보였으나 학습 곡선 때문에 MVP 단계는 gluestack 우선.)

**참고 소스 (테마 토큰 잡을 때)**
- 당근 **Seed Design** — 토큰·컬러·타이포 가이드 참고용
- 토스 오픈소스(`es-toolkit`, `overlay-kit` 등) — UI는 아니지만 로직/오버레이 유틸로 유용

**토스st 룩 레시피 (테마 토큰 기준)**
- **라운드**: 버튼·카드·인풋 12~16px로 큼직하게
- **여백**: 요소 간 간격 넉넉히, 화면이 숨 쉬게
- **타이포**: 제목 bold / 본문 regular 대비 분명, 자간 살짝 좁게
- **컬러**: 메인 컬러 1개 + 회색 다단계, 채도 낮은 배경 + 선명한 포인트
- **그림자**: 큰 blur · 낮은 opacity로 부드럽게
- **폰트**: **Pretendard**(오픈소스, 토스st 느낌의 핵심)

> 게임화 연출(상자 개봉·잭팟)은 디자인 시스템이 제공하지 않으므로 `reanimated` + `lottie`로 별도 구현.

### 4.2 기프티콘 교환 관련 기술
사업자등록 보유 전제. 한국 시장 B2B 기프티콘 발송 API 연동.

- **연동 방식**: 운영자가 발급사에 **선충전(예치금)** → 사용자가 캐시로 교환 요청 → 서버가 발급사 API 호출 → 기프티콘(MMS/링크) 자동 발송
- **검토 대상 발급사(예시, 실제 계약 시 조건 비교 필요)**
  - 기프티쇼 비즈(Bizz)
  - 쿠프마케팅
  - 윈큐브마케팅
  - (그 외 카카오/스마일콘 계열 B2B 채널)
- **선정 시 비교 항목**
  - 상품 라인업(편의점/카페/상품권 등)
  - 건당 수수료 / 최소 선충전금
  - 발송 성공률·재발송 처리
  - 정산/세금계산서, API 안정성, 사용량 한도
- **구현 시 주의**
  - 교환은 서버에서 **원자적 트랜잭션**으로: 캐시 차감 ↔ 발급 API 성공이 함께 성립해야 함. 발급 실패 시 캐시 롤백.
  - 멱등성 키(idempotency key)로 중복 발급 방지
  - 발급 결과/실패 로그 보관, 사용자 교환 내역 화면 제공

### 5.2 사용자 / 캐시 / 학습 도메인

> **설계 원칙 (확정)**
> - DB: **PostgreSQL**. 캐시는 자산이므로 정합성 최우선.
> - 캐시 적립/사용은 **단일 통합 원장(`ledger`)** 에 +/- 로 기록하되, `direction`(earn/use) 컬럼으로 화면 구분.
> - **잔액(`wallet.balance`)은 원장의 합과 항상 일치**해야 하며, 모든 캐시 변동은 *원장 기록 + 잔액 갱신*을 **하나의 트랜잭션**으로 처리.
> - 모든 캐시 관련 유저 행위(회원/지갑/거래/퀴즈/데일리)는 빠짐없이 DB에 기록.

#### 관계 요약
- `user` 1 ─ 1 `wallet`
- `user` 1 ─ N `ledger` / `quiz_log` / `cash_box` / `srs_state` / `attendance` / `gift_exchange` / `ad_reward_log`
- `user` 1 ─ N `daily` (일자별)
- `ledger.ref_id` → 거래를 발생시킨 원본 엔티티(`cash_box`, `gift_exchange`, `attendance` 등)로 역추적
- `quiz_log.box_id` → `cash_box` (정답으로 생성된 상자)

#### 정합성 체크(운영 시 정기 검증 권장)
```
-- 잔액 = 적립합 - 사용합 인지 검증
SELECT u.id,
       w.balance,
       COALESCE(SUM(CASE WHEN l.direction='earn' THEN l.amount
                         WHEN l.direction='use'  THEN -l.amount END), 0) AS ledger_balance
FROM "user" u
JOIN wallet w ON w.user_id = u.id
LEFT JOIN ledger l ON l.user_id = u.id
GROUP BY u.id, w.balance
HAVING w.balance <> COALESCE(SUM(CASE WHEN l.direction='earn' THEN l.amount
                                      WHEN l.direction='use'  THEN -l.amount END), 0);
-- 결과가 0행이어야 정상
```

## 6. 데이터 정제 파이프라인 (선행 작업)

보유 데이터를 서비스에 쓰려면 아래 정제가 선행되어야 함. Python(Django/스크립트)으로 처리 권장.

### 6.1 단어 타입 구분 (kana / kanji)
- `surface`에 한자 포함 여부로 분류 → `word_type` 채우기
- 일본어 문자 범위(한자 U+4E00–U+9FFF 등) 정규식으로 판별

### 6.2 품사 HTML 파싱
- raw HTML 저장된 `pos` → 태그 제거·정규화 → enum/표준 라벨로 매핑
- 표준 품사 라벨 셋 정의(명사/동사(자·타)/い형용사/な형용사/부사/조사 등)
- 오답 생성 시 이 표준 품사를 키로 사용

### 6.3 한국어 뜻 형식 정규화
불일치 예시:
```
頭 : 1. 머리 2. 두부 3. 두발
いい : 1. 좋다 2. '나쁘다·꼴사납다'의 뜻을 나타냄
十時 : 10시
四杯 : 네 잔, 넉 잔
~月 : …월
```
처리 방침:
- "1. … 2. …" 패턴 → 의미 단위로 분해해 `word_meaning`에 sense_no별 저장
- 단일 뜻은 sense_no=1로 저장
- "쉼표/가운뎃점"으로 나열된 동의어는 한 sense 내에서 유지하거나 정책에 따라 분할
- 부가설명("~의 정중한 말" 등)은 `note`로 분리
- 정규화 규칙은 룰베이스 + 애매한 케이스는 AI 보정 병행

### 6.4 JLPT 레벨 AI 태깅 (단어)
- 단어 `jlpt_level`이 비어있음 → AI로 N5~N1 추정 태깅
- 검증: 표준 JLPT 어휘 리스트(보유한 급수별 데이터)와 **교차 매칭**으로 정확도 보정
- 신뢰도 낮은 항목은 플래그 → 수동 검수 큐로

### 6.5 정제 산출물 검증
- 오답 생성 가능성 확인: "각 (급수×품사) 버킷에 distractor로 쓸 후보가 충분한가" 점검
- 버킷이 빈약하면 오답 추출 규칙 완화(인접 급수 허용 등)

## 7. 보유/추가 데이터 검토

### 보유 (확인됨)
- 단어 약 7,000개 / 한자 약 2,200자 / JLPT 급수별 데이터

### 있으면 좋은 추가 데이터(향후)
- **예문**(단어별 일본어 예문 + 한국어 번역) → 학습 깊이·문맥 학습
- **발음 음성(TTS 또는 음원)** → 듣기 퀴즈 확장
- **단어 빈도(코퍼스 빈도)** → 출제 우선순위·중요 단어 강조
- **한자 획순 데이터** → 한자 쓰기 학습(2차)
- **연어/관용구** → 고급 학습 콘텐츠

## 8. 단계별 기능 로드맵 (MVP / 2차 / 3차)

### 🟢 MVP (1차 출시) — "핵심 루프가 돈다"
출시에 반드시 필요한 최소 기능. 목표: *퀴즈 풀고 → 캐시 모으고 → 기프티콘으로 바꾼다*가 끝까지 작동.

**학습**
- 구글 소셜 로그인 + 온보딩(JLPT 급수 선택)
- 단어 퀴즈 / 한자 퀴즈 (4지선다, 단어↔뜻)
- 오답 생성(동일 품사·급수 유사 항목)
- SRS 출제(복습 우선 + 신규 혼합)

**리워드**
- 정답 시 캐시상자 획득 → 인벤토리 적립
- 상자 개봉(3개당 AdMob 보상형 광고 1회) + 등급별 캐시 지급
- 지갑(잔액) + 캐시 적립/사용 내역(`ledger`)
- 출석 체크 + 연속 출석 보너스

**교환/수익**
- 기프티콘 교환소(B2B API 연동, 교환 시 광고)
- AdMob 보상형 광고 + **SSV 서버 검증**

**안정성(보이지 않지만 필수)**
- 캐시 정합성(원장+잔액 트랜잭션)
- 일일 상한 + 기본 어뷰징 방어(서버 검증, 응답속도/패턴 체크, Play Integrity)
- 데이터 정제 파이프라인 완료(타입/품사/뜻/JLPT 태깅) — *MVP 이전 선행*

> **MVP에서 빼는 것**: 잠금화면, 예문/음성, 입력형·듣기 퀴즈, 랭킹/소셜, 추가 광고 지점(보너스 상자), iOS.

---

### 🟡 2차 — "리텐션과 학습 깊이를 키운다"
MVP가 돌아가고 사용자가 붙기 시작하면 추가.

- **잠금화면 학습** (부제에 명시한 기능) — 잠금화면 위젯/오버레이로 단어 노출 + 간단 퀴즈, 학습 시 소액 캐시
- **광고 보너스 상자** — "광고 보고 상자 1개 더"(1일 N회 한도) → 수익·리텐션 동시 강화
- **예문 학습** — 단어별 일본어 예문 + 번역(데이터 확보 시)
- **듣기 퀴즈** — TTS 음성 → 단어 맞히기
- **푸시 알림** — 출석 리마인드, 복습 기한 알림(FCM)
- **스트릭 보호권** — 광고 시청으로 연속 출석 1회 방어
- **학습 통계 화면** — 정답률·약점 단어·학습량(daily 집계 활용)
- **난이도별 차등 지급** 본격 적용 + 환율/확률 A/B 튜닝

---

### 🔵 3차 — "확장과 차별화"
규모가 커진 뒤의 확장.

- **iOS 출시** (RN이라 확장 용이)
- **입력형 퀴즈**(히라가나 타이핑), **한자 획순 학습**(획순 데이터 확보 시)
- **랭킹/리더보드, 친구 초대 리워드**(소셜 + 바이럴)
- **테마/꾸미기**(상자 스킨 등) — 인앱 재미 요소
- **고급 콘텐츠** — 연어·관용구, 빈도 기반 중요 단어 코스
- **추가 수익 모델 검토** — 광고 제거 구독, 프리미엄 학습팩 등
- **운영 도구 고도화** — 어뷰징 대시보드, 의심 계정 로그 장기 보존

- iOS 확장 시점
